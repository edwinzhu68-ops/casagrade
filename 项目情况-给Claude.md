# 彩票项目当前主要情况（复制给 Claude 用）

## 一、项目是什么

- **巴拿马彩票（Loteria）** 的销售与开奖系统：顾客在客户端选号下单，门店老板确认收款、兑奖，总后台拉取官方开奖结果并手动发送开奖、做结算。
- **两个代码库**：
  - **lottery-preview**：前端，静态 HTML + JS，页面有客户端、老板端、总后台、开奖结算页。
  - **lottery-system**：后端，NestJS，提供订单、店铺、开奖、结算、管理员等 API；API 基地址一般为 `https://api.casagrade.com`。

---

## 二、前端页面与角色

| 文件 | 角色 | 作用 |
|------|------|------|
| **index.html** | 顾客 | 输入店号进店 → 选 Billete($1) / Chance($0.25)、号码与数量 → 提交订单 → 付款页（二维码）→ 轮询到已付款后进已确认页，可看开奖结果与中奖金额 |
| **merchant.html** | 老板/门店 | 登录收银台 → 看本期销售额、热门号码、未付款/已付款订单 → 确认收款、兑奖 → 可跳转 result.html 看结算 |
| **dashboard.html** | 总管理员 | 登录 → 拉取 Firebase 官方开奖 → 填 1/2/3 奖号码 → 发送开奖结果 → 分配店号、查销售明细等；有「清空开奖结算」按钮（登录即可，不需管理员密钥） |
| **result.html** | 老板/门店 | 开奖结算页：当前期销售额、赔付、利润、订单数/中奖订单数、订单列表（按订单号后四位排序，已中奖黄/未中奖灰）、Billete/Chance 中奖汇总、最近 20 期历史（可点进某期详情） |

## 三、业务与期次逻辑（已和产品确认）

- **销售期**：上一期开奖后 16:00 起售，本期开奖日 14:55 停售，15:00 开奖；开奖后数据进结算页，16:00 销售页清除已开奖期数据、开始新一期。
- **结算页**：开奖当天 15:00 前「当前期」在展示上清空（转历史），等 15:00 开奖后新数据注入；**不需要 9 点定时任务**。总后台可手动点「清空开奖结算」实现同样效果（当前期归档，结算页显示等待开奖）。
- **后端**：Draw 表有 `archived_at`；`GET /api/draw/latest` 只返回未归档的最近已开奖期；`POST /api/admin/clear-settlement` 把最近已开奖期设 `archived_at`，登录即可调（不校验管理员密钥）。

## 四、中奖规则（与后端一致）

- **Billete**：4 位号，$1/张。与 1/2/3 奖各取**最高档**，三奖金额**相加**。档位从高到低：四位全中 [2000,600,300] → 前3/后3 [50,20,10] → 前2 [3,0,0] → 后2 [3,2,1] → 后1 [1,0,0]。
- **Chance**：2 位号（或取后两位），$0.25/张。只看三个开奖号的**后两位**，[14, 3, 2] 元/张，**三奖可叠加**，最多 19 元/张。结算必须「张数×赔率」，不能按金额×张数。
- 详细规则见项目内 **中奖规则.md**。

---

## 五、重要约定与易错点

1. **中奖金额以后端为准**：前端统一用 `win_amount`，为 0 时用 `win_breakdown` 各项之和兜底，不在前端按赔率重算。
2. **两条结算路径**：开奖时 **DrawController.settleOrdersForDraw** 写订单的 win_amount/win_breakdown；**SettlementService** 做期次汇总，Chance 必须按张数×[14,3,2]，两处逻辑与 dist 编译结果都要一致。
3. **编译一致性**：曾因 dist 与 src 不一致导致 CHANCE 被按 80 元/张算错；改赔率或结算后需重新 build，部署前建议清 dist 再编译。
4. **路由**：`POST /api/admin/clear-settlement` 在 **draw.controller.ts** 的 AdminController 里也有注册（与 modules/admin 的 AdminController 可能并存），避免出现 Cannot POST。

## 七、当前技术栈与结构

- **前端**：无构建，直接 HTML + 内联或脚本，Tailwind CDN，部分页面西语/中文。
- **后端**：NestJS + TypeORM，SQLite，entities：Order、Shop、Draw、User；模块：Order、Draw、Merchant、Settlement、Admin。
- **鉴权**：商户/管理员登录用 `POST /api/merchant/login`；总后台部分操作需请求头 `X-Admin-Token`（与服务器 ADMIN_TOKEN 一致），「清空开奖结算」和健康检查不需要。
