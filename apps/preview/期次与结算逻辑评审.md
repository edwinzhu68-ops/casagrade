# 期次与结算逻辑评审

按你确认的「销售页 / 结算页 / 9 点 / 清空」逻辑整理，并标出与实现的对照与改进点。

---

## 一、销售页 vs 结算页（你定的逻辑）

**销售页面（老板端本期销售额、热门号码等）：**

- 统计的是**开奖后当期的销售数据**。
- 数据从**上一期开完奖后的 16:00** 开始算，一直到**本期的开奖日 15:00** 停售。
- **15:00（3 点）开奖后**，销售页这一期的所有数据**转移到结算页面**（结算页接收开奖结果 + 订单数据）。
- **16:00** 即**清除已开奖期的销售数据**，开始售卖新一期，等待新一期数据（不再保留到当晚）。

**结算页面（result 开奖结算）：**

- 开奖后，**当前期的订单与统计**注入结算页（销售额、中奖、利润、订单列表等）。
- **开奖当天 15:00 前**：结算页**清空**（当前期转为历史），以便 15:00 开奖后有新数据注入；**不需要 9 点定时任务**，只要 15:00 前清除即可。
- 注意：这里说的「清空」是**把结算页当前期变成历史**，不是清空「销售界面」；销售界面在 16:00 清除已开奖数据、开始新一期。

## 二、生命周期（按你逻辑整理）

```
上一期 16:00 起 → 销售页清除已开奖数据、统计新一期销售 → 本期 15:00 停售
→ 开奖当天 15:00 前 结算页清空（当前期转历史），等开奖后数据注入
→ 15:00 开奖 → 数据注入结算页（销售页本期数据也转到结算页）
→ 16:00 销售页清除已开奖期数据，开始新一期
```

**当前实现对照：**

| 环节 | 你的描述 | 当前实现 | 建议 |
|------|----------|----------|------|
| 开奖 | 15:00 开奖 | 总后台手动填 1/2/3 奖点「发送」；时间可配 | 若需严格 15:00 可加「到点自动拉 Firebase」 |
| 开奖期 | 15:00–16:00 约 1 小时 | bet-status 有 isDrawWindow，前端 15:00 停售、开奖中提示 | 一致 |
| 销售页 → 结算页 | 3 点后本期数据转到结算页 | 开奖后 settleOrdersForDraw，结算页用订单聚合 | 一致 |
| 结算页清空 | 开奖当天 15:00 前当前期转历史，不需 9 点定时 | 有 settlement/history 最近 7 期 | 需实现：15:00 前展示上归历史、结算页空出等新数据；不需定时任务 |
| 销售数据 16:00 清 | 16:00 清除已开奖期销售数据，开始新一期 | 老板端按 draw 统计 | 16:00 起只展示新期 draw，已开奖期不再算「本期」即可 |
| 发送开奖结果 | 总后台点发送 | POST /api/draw/manual → settleOrdersForDraw | 一致 |

## 三、结算页「当前期」展示内容

- 当前期汇总：销售额、中奖金额、利润、订单数量、中奖订单数量  
- 中奖统计：Chance / Billete 分块，每号「张数 → 赔 $xxx」，再总赔付  
- 所有订单：开奖后全部订单在结算页，按**订单号后四位**排序；**样式与客户端一致**：已中奖黄色，未中奖灰色；每条：订单号、店铺、购买号码、金额、中奖金额、状态；点订单看完整投注+中奖详情  

**当前 result.html 对照：**

- 有：销售额、Chance/Billete 销售额、总赔付、利润、已付款订单列表、Billete/Chance 中奖号码汇总（号码×张数×应赔）、订单详情弹窗。  
- 缺/需加强：  
  - **订单数量、中奖订单数量** 两个数字可显式标出。  
  - **订单列表按订单号后四位排序**：按现有逻辑即可；前端排序 `order_number.slice(-4)`。  
  - **订单行颜色**：与客户端一致——已中奖黄色，未中奖灰色。  
  - **每条订单显示「店铺」**：若当前只显示订单号+金额+状态，可加 shop_number/shop_name（需接口或列表带店铺信息）。

## 四、结算页「开奖当天清空」说明（已按你纠正）

- **清空的是结算页**，不是销售界面。  
- **开奖当天 15:00 前**，结算页把当前展示的那一期**转为历史**（数据不删），结算页界面清空，以便 15:00 开奖后**有新数据注入**；开奖结果发送后，结算页再显示本期的销售额/赔付/订单/中奖统计。  
- 实现上：不删库，用 draw 状态（pending / completed）和「当前期 / 历史」的展示规则即可；**不需要 9 点定时任务**，只要开奖当天 15:00 前清除即可。

## 六、开奖触发与结算页清空（已按你确认）

- 总后台点「发送」后：结算所有订单 → 结算页计算中奖、生成统计；结算页数据保留到**下一期开奖日**。
- **开奖当天 15:00 前**：当前期在展示上**转为历史**，结算页清空，等 15:00 开奖后新数据注入。**不需要 9 点定时任务**，反正开奖当天 15:00 前清除就行。

**实现上（大白话）：**

- 不删库，订单和 draw 都保留。
- 「当前期」和「历史」只做**展示上的区分**：开奖当天、且尚未发送开奖结果前，结算页当前期可视为历史/空着，显示「等待开奖」；发送后显示本期。不需定时任务。
- 历史列表：按 draw 倒序取最近 20 期即可。

## 八、建议的文档约定（写入运行逻辑）

1. **期次**：一期 = 一个 draw（pending → completed）；15:00 开奖、15:00–16:00 开奖期、16:00 起售下一期。  
2. **当前期**：最近一次 status=completed 的 draw；结算页展示该期销售额、赔付、利润、订单数、中奖订单数、订单列表（按订单号后四位排序）、Chance/Billete 中奖统计。  
3. **历史**：过去 N 期（如 20 期），每期摘要；点进某期可看该期全部订单、中奖统计、号码统计。  
4. **数据**：不删库；订单与 draw 永久保留；仅通过 draw 状态与「当前期/历史」展示规则区分。  
5. **结算页清空**（已定）：**开奖当天 15:00 前**当前期在展示上归入历史，结算页清空等开奖后数据注入，不需 9 点定时任务；**销售页 16:00** 清除已开奖期数据，开始新一期。
