# 协作说明

本文档记录项目已实现的改动与约定，便于协作与后续维护。

## 一、已实现改动汇总

### 1. 前端（lottery-preview）

#### 1.1 总后台 dashboard.html
- **开奖日期更新**：发送开奖结果后，开奖日期会从待开奖接口拉取并更新为下一期（如 3/15 → 3/18）。
  - 增加 `normalizeDrawDateToYMD()`，支持后端返回的 YYYY-MM-DD、DD-MM-YYYY 及 `draw_date` / `drawDate` 等字段。
  - 手动开奖成功且结算完成后，延迟 400ms 再调 `initDrawDefaults()`，确保拉到新创建的下一期。

#### 1.2 开奖结算页 result.html
- **中奖金额展示统一**：优先使用后端 `win_amount`，若为 0 则用 `win_breakdown` 各项之和，避免「前端算不准」。
  - 新增 `getOrderWinAmount(order)`，总赔付、订单列表、订单详情均用该函数显示金额。
  - `PAYOUTS` 仅作展示参考，注释标明「实际金额以后端结算为准」。

#### 1.3 客户端 index.html
- **开奖号码展示**：在付款页、已确认页的二维码下方增加「Resultado del sorteo」，显示头奖、二奖、三奖开奖号码（从 `/api/draw/latest` 拉取）。
- **中奖金额展示**：从 API 更新已付款列表时，若 `win_amount === 0` 且存在 `win_breakdown`，用 breakdown 之和写回展示；列表展示用 `getDisplayWinAmount(info)` 做同样兜底。

#### 1.4 老板端 merchant.html
- **中奖金额展示**：新增 `getOrderWinAmount(order)`，待兑奖列表、待兑付总额、订单详情弹窗、确认兑奖均用该函数，与后端一致。

### 2. 后端（lottery-system/backend）

#### 2.1 中奖规则（draw.controller.ts）
- **BILlete**：每奖只取最高档，三奖相加；赔率表与数据库规则一致（4位全中 [2000,600,300]，前3/后3 [50,20,10]，前2 [3,0,0]，后2 [3,2,1]，后1 [1,0,0]）。
- **CHANCE**：只看一二三等奖的**后两位**，[14, 3, 2] 元/张，三奖可叠加；投注号统一 `num.slice(-2)`，不论填 2 位还是 4 位。

#### 2.2 结算与编译一致性（重要）
- **订单级结算**：`draw.controller.ts` 中的 `settleOrdersForDraw` 在开奖完成时结算本期已付款订单，写入 `win_amount`、`win_breakdown`、status。
- **SettlementService**：另有 `settlement.service.ts` 的结算逻辑；Chance 赔付应按**张数 × 赔率**（如 14×qty、3×qty、2×qty），不可用「金额×张数」(betAmount) 计算。
- **编译与运行一致**：曾出现 **dist 与 src 不一致** 导致 CHANCE 被错误按 80 元/张（Billete first3）计算。修复方式：重新编译并重启；建议部署前清空 dist 再 build，必要时在启动时校验 `CHANCE_RATE === [14, 3, 2]`。

### 3. 历史问题与结论

- **订单 MMR5EN1D2951（78×122 张等）显示 9760/17040**：根因是当时**编译产物中 CHANCE 常量错误为 80**，导致按 80 元/张计算；修正 dist 与源码一致并重启后已恢复正确。
- **两条结算路径**：DrawController（开奖时）与 SettlementService 均可能参与结算；排查赔付错误时需同时检查两处逻辑及 dist 是否与 src 一致。

## 二、约定与注意事项

1. **中奖金额以后端为准**：前端仅展示 `win_amount`，若为 0 则用 `win_breakdown` 之和兜底，不在前端用赔率重算。
2. **Chance 规则**：仅后两位、[14, 3, 2]、三奖叠加，最高 19 元/张；Billete 为每奖最高档、三奖相加。
3. **后端发布**：修改赔率或结算逻辑后务必重新编译并确认 dist 与 src 一致后再部署。
