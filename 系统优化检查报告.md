# 系统优化检查报告

从安全、健壮性、可维护性与生产就绪度等角度对彩票系统做的检查与已实施/建议优化。

---

## 一、已实施的优化（含报告建议项）

### 1. 后端 API 规范与安全

- **下单入参校验（POST /api/orders）**
  - 必填：`shopId` 存在且有效。
  - `numbers`：数组长度 1～500，每项 `n` 字符串长度 ≤10，`q` 为 1～999。
  - `amount`：大于 0、不超过 100000。
  - 不通过时返回 `400 Bad Request` 及明确文案，避免脏数据与滥用。

- **列表查询 limit 上限**
  - `GET /api/shop/:shopId/orders` 的 `limit` 参数上限为 500，避免单次拉取过多订单影响性能与内存。

- **HTTP 状态码与异常**
  - 订单/店铺不存在 → `NotFoundException`（404）。
  - 业务规则不满足（超时、状态不对、已兑奖等）→ `BadRequestException`（400）。
  - 登录/鉴权失败 → `UnauthorizedException`（401）。
  - 全局异常过滤器：未显式抛出的 `Error` 统一返回 500 且将 `message` 透出，便于排查；`HttpException` 仍按 Nest 默认返回对应 status 与 body。

### 2. 前端错误文案

- 后端 4xx 返回体中 Nest 使用 `message` 字段存放中文提示。顾客端下单、老板端登录、总后台登录/店铺对比等处已统一为优先使用 `data.message || data.error` 再展示给用户，避免只看到 "Bad Request" 等英文。

### 3. 管理员接口鉴权（已实施）

- 环境变量 **ADMIN_TOKEN**：若设置，则以下接口必须在请求头带 **X-Admin-Token** 且与该值一致才能访问：
  - `POST /api/draw/time`、`POST /api/draw/manual`
  - `POST /api/admin/draw`
  - `POST /api/settlement/settle/:drawId`
  - `GET /api/admin/shop-compare`
- `GET /api/admin/health` 不校验，便于探活。
- 总后台（dashboard）登录页已增加「管理员密钥」输入框，登录时写入本地并在此后请求上述接口时自动带上；未配置 ADMIN_TOKEN 时后端不校验，兼容本地开发。

### 4. CORS 白名单（已实施）

- 环境变量 **ALLOWED_ORIGINS**：多个来源用英文逗号分隔（如 `https://a.com,https://b.com`）。设置后仅允许这些 origin；不设置则允许所有来源。

### 5. 数据库配置（已实施）

- **DATABASE_PATH**：数据库文件路径，默认 `lottery.db`。
- **NODE_ENV=production** 时关闭 **synchronize**，避免生产自动改表；非 production 仍为 true 便于开发。
- 项目根目录提供 **.env.example**，部署时可复制为 `.env` 并按需填写；已使用 `dotenv/config` 在启动时加载。

### 6. 密码校验（已实施）

- 老板端登录：若库中 `password_hash` 为 bcrypt 格式（`$2a$`/`$2b$`/`$2y$`），则用 **bcrypt.compare** 校验；否则仍兼容原 SHA256 或明文，便于旧数据过渡。新用户建议用 bcrypt 存库。

## 二、建议优化（后续可选）

### 1. 安全

| 项 | 说明 | 优先级 |
|----|------|--------|
| **老板端 Token** | 当前为 Base64(userId:account)，无签名与过期。可选：JWT 或 signed token + 短期过期；敏感操作可要求重新登录或刷新 token。 | 中 |
| **兑奖/确认幂等** | 兑奖、确认收款等若因重试导致重复请求，当前依赖状态判断（已兑奖/已付款会报错）。可考虑：兑奖接口对同一订单返回相同成功结果，避免前端重复点击产生歧义。 | 低 |
| **订单号碰撞** | 订单号生成若仅用时间戳+随机数，高并发下理论上有极小碰撞概率。可加 DB 唯一约束 + 重试一次生成。 | 低 |

### 2. 数据与配置（其余）

| 项 | 说明 | 优先级 |
|----|------|--------|
| **生产表结构** | 生产环境已关闭 synchronize；若后续改表，可用 TypeORM 迁移或手动 SQL。 | 低 |

### 3. 可维护性与运维

| 项 | 说明 | 优先级 |
|----|------|--------|
| **请求体校验** | 使用 `class-validator` + `ValidationPipe` 对 DTO 做声明式校验，减少手写 if 与统一错误格式。 | 低 |
| **日志** | 关键操作（下单、确认、兑奖、开奖）已有 log；可补充请求 ID、耗时，便于排查与监控。 | 低 |
| **健康检查** | `/api/admin/health` 已存在；若部署在 K8s/负载均衡后，可保证该路径不被鉴权拦截以便探活。 | 已具备 |

### 4. 前端

| 项 | 说明 | 优先级 |
|----|------|--------|
| **动态内容转义** | 使用 `innerHTML` 插入订单号、金额等时，若数据来源不可控，建议对展示内容做转义或使用 `textContent`/仅插文本，防止 XSS。当前订单号由后端生成、格式可控，风险较低，可后续统一规范。 | 低 |
| **API 错误统一展示** | 部分请求仅 `alert(data.message)`，可统一为 toast 或页面内提示，并区分 4xx/5xx。 | 低 |

## 三、与「上线前自测清单」的对应关系

- 自测清单已覆盖：顾客下单、老板确认、兑奖、开奖、多单、语言、输入订单号等核心流程。
- 本报告中的「已实施优化」不影响既有流程，仅加强参数校验、错误码与 limit 上限；自测时若遇「缺少店铺ID」「金额无效」等提示，多为故意非法请求，属正常防护。
- 管理员鉴权与 CORS 已实施；上线时在服务端配置 `ADMIN_TOKEN`、`ALLOWED_ORIGINS` 等即可，其余按排期可选。

---

## 四、小结

- **已完成**：下单入参校验、订单列表 limit 上限、HttpException 与全局异常过滤、前端错误文案、**Admin 鉴权（X-Admin-Token + 环境变量）**、**CORS 白名单**、**数据库路径与 synchronize 环境变量**、**老板端登录 bcrypt 校验（兼容旧密码）**、`.env.example` 与 dotenv 加载。
- **后续可选**：老板端 Token 升级（JWT/过期）、请求体校验与日志增强、前端 XSS 与幂等策略。

**不影响使用**：不配置任何环境变量时，和以前一样照常用；总后台开奖/结算、老板端登录、顾客下单等都不受影响。  
**需要提高安全性时**：在服务器上设置 `ADMIN_TOKEN`（总后台登录时填同一密钥）、`ALLOWED_ORIGINS`（前端域名）等即可，详见 `lottery-system/backend/.env.example`。
