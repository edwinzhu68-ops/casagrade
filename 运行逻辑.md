# 彩票系统运行逻辑（供 OpenClaw 查 Bug 用）

本文档描述客户端、老板端、总后台、开奖结算页及后端的完整运行逻辑与数据流，便于排查遗漏或错误。

---

## 一、系统概览

- **前端**：静态 HTML + JS，API 基地址一般为 `https://api.casagrade.com`（可本地覆盖）。
- **后端**：NestJS，提供订单、店铺、开奖、结算、管理员等接口。
- **角色**：顾客（客户端）、门店/老板（老板端）、总管理员（总后台）。

---

## 二、客户端（index.html）

### 2.1 入口与进店
- 打开页面：首页为输入店号（1～9 位数字）。
- 可选「Cuenta」进行本地账号注册/登录（与订单关联的 clientId / 展示名）。
- 点击「Entrar」：请求 `GET /api/shop/:shopNumber` 获取店铺信息；请求 `GET /api/bet-status` 获取是否可售、是否开奖窗口、本期日期等。
- 若可售，进入店铺页（page-shop），显示店号、本期开奖日期、玩法选择（Billete / Chance）。

### 2.2 下单与停售
- 选择 Billete（$1/张）或 Chance（$0.25/张），输入号码（Billete 4 位、Chance 2 位）和数量，加入列表，可多组号码。
- 页面会轮询 `GET /api/bet-status`：开奖窗口内（如 14:55～16:00 巴拿马时间）禁止下单并显示「Sorteo en curso」；临近停售显示倒计时。
- 点击提交：`POST /api/orders` 传 `shopId, numbers: [{n, q}], amount, gameType, clientId`。后端会取当前待开奖期 `draw_id` 写入订单。

### 2.3 付款流程
- 下单成功后进入付款页（page-pay）：显示订单号、金额、二维码（含 orderId/orderNumber/orderHash/amount/shopId）、店号。
- 二维码下方会拉取 `GET /api/draw/latest` 显示「Resultado del sorteo」头/二/三奖号码（有则显示）。
- 30 分钟倒计时；轮询 `GET /api/orders/:orderNumber` 检测 `status === 'paid'`。一旦变为已付款：停止轮询、跳转已确认页、订单写入本地「已付款列表」并拉取开奖状态更新（含 win_amount / win_breakdown 兜底）。

### 2.4 已确认/订单详情
- 已确认页（page-confirmed）：同一二维码、店号、订单号、金额、号码明细；下方同样拉取最新开奖 1/2/3 奖显示。
- 根据订单状态展示：等待开奖 / 已开奖未中 / 已中奖（金额） / 已兑奖。中奖金额优先用接口 `win_amount`，为 0 时用 `win_breakdown` 之和。
- 店铺页有「已付款订单」列表（来自本地 + 接口更新）；点击某单进入同一已确认页并带 token 拉取最新订单详情与开奖结果。

### 2.5 数据与状态
- 未付款订单存 `localStorage`（clientId + 店号等），付款后移除并写入已付款列表；已付款列表会按接口更新 status、win_amount、redeemed_at 等。
- 开奖日期、是否可售、倒计时均来自 `GET /api/bet-status`，与后端待开奖期一致。

## 三、老板端（merchant.html）

### 3.1 登录与店铺
- 输入账号（字母+数字）或店号（纯数字）+ 密码，`POST /api/merchant/login`；成功后拉取 `GET /api/merchant/shops` 取当前店铺，进入收银台（page-cashier）。
- 可设置 Chance/Billete 限额与单价（存 localStorage），用于展示与风控提示。

### 3.2 仪表盘与本期统计
- 轮询/加载：`GET /api/shop/:shopId/orders`、`GET /api/draw/pending`、`GET /api/draw/latest`。
- **本期订单**：`periodOrders` = 属于当前待开奖期（pending draw_id）的订单；无待开奖期时用 `draw_id == null` 的订单。
- **本期已付款**：`ordersForCurrentDraw` = 本期订单里 status 为 paid(1) 的，用于今日销售额、今日收入、热门号码统计。
- **开奖后**：`drawCompleted = true` 时不再用「全部已付款」兜底，热门号码只基于本期已付款，避免历史数据干扰。
- 待兑奖：status=won(3) 且 redeemed_at 为空；待兑付金额用 `getOrderWinAmount(order)`（win_amount 或 win_breakdown 之和）汇总。

### 3.3 未付款订单与确认收款
- 未付款订单：status=pending(0)，且创建时间在 30 分钟内；可按订单号尾数筛选。
- 点击「确认」：`POST /api/orders/:orderNumber/confirm`，body `{ shopId }`。后端将 status 置为 1(paid)，若有需要会补写 `draw_id` 为当前待开奖期。
- 客户端付款页轮询到 paid 后会自动跳转已确认并更新本地列表。

### 3.4 兑奖
- 输入订单号或后几位，请求 `GET /api/shop/:shopId/orders?status=won&suffix=xxx` 或 `GET /api/orders/:orderNumber` 匹配已中奖未兑奖订单。
- 选择订单后调用 `POST /api/orders/:orderNumber/redeem`，body `{ shopId }`。后端校验：订单属于该 shopId、status=3、redeemed_at 为空，然后写入 redeemed_at。
- 订单详情弹窗、待兑奖列表的中奖金额均用 `getOrderWinAmount(order)` 显示。

### 3.5 本期订单与订单详情
- 「本期订单」列表：展示 periodOrders 中已确认（非 pending）的订单；点击整行或单号打开订单详情弹窗（z-index 低于订单详情，订单详情为 z-[60]）。
- 订单详情展示号码、金额、状态、中奖金额及 win_breakdown（若有）。

### 3.6 开奖结算页入口
- 从收银台跳转 `result.html?t=token&sid=shopId&sn=shopNumber&sname=shopName`，带 token 与店铺信息，用于查看该店开奖结果与结算数据。

## 四、总后台（dashboard.html）

### 4.1 登录
- 管理员账号+密码 `POST /api/merchant/login`；可选填管理员密钥（与服务器 ADMIN_TOKEN 一致），用于开奖/结算等敏感操作。

### 4.2 统计与数据
- 拉取店铺数、今日订单、今日销售额等（具体接口依实现，可能为 admin 或聚合接口）。
- 可按店号查询某店销售明细（Chance/Billete 销量分布）。

### 4.3 Firebase 开奖拉取
- 「拉取一次」：`GET /api/draw/fetch-firebase`（需 AdminToken）从巴拿马官方拉取开奖数据，返回 1/2/3 奖号码等，供填入手动开奖表单。
- 「开始轮询」：定时拉取，拉取到完整号码后可自动填入；日期由下方手动开奖处单独选择。

### 4.4 手动开奖
- **开奖日期/时间**：从 `GET /api/draw/pending` 初始化（支持 YYYY-MM-DD、DD-MM-YYYY 及 draw_date/drawDate）；手动可改，用于轮询时间或发送开奖时间。
- **发送开奖时间**：`POST /api/draw/time` 更新待开奖期的 draw_date/draw_time（若有）。
- **发送开奖结果**：填写 1/2/3 奖号码，点击后 `POST /api/draw/manual`，body 含 `billete/segundas/terceras`（或 primer/segundo/tercero）及 drawTime。后端会：
  - 将当前待开奖期标记为 completed，写入 winning_numbers；
  - 调用 **settleOrdersForDraw** 结算本期（draw_id = 该期）所有 status=1 的订单，写入 win_amount、win_breakdown、status(2/3)；
  - 创建下一期待开奖（若不存在）；
- 前端在手动开奖成功、以及结算完成后延迟再调 `initDrawDefaults()`，从 pending 拉取下一期日期，使「开奖日期」更新为下一期（如 3/18）。

### 4.5 结算
- 总后台在发送开奖结果后，会再请求 `GET /api/draw/latest` 取 drawId，然后 `POST /api/settlement/settle/:drawId`（需 AdminToken）做汇总结算（SettlementService）。注意：**订单级中奖金额已在 DrawController 的 settleOrdersForDraw 中写入**；SettlementService 做的是期次维度汇总或报表，其 Chance 赔付必须按「张数×赔率」计算，不能按「金额×张数」(betAmount)。

### 4.6 分配店号
- 输入店号与买家账号，`POST /api/admin/assign-shop`，将店号绑定到该账号（用于老板端用店号登录）。

### 4.7 清空开奖结算
- **清空开奖结算**按钮：`POST /api/admin/clear-settlement`（需 AdminToken）。效果与「开奖当天 15:00 前清除」一致：将当前期（最近一次已开奖）归档，结算页 `GET /api/draw/latest` 不再返回该期，页面显示「等待开奖」直到下次发送开奖结果；数据不删，该期仍在历史记录中可查。

## 五、开奖结算页（result.html）

### 5.1 进入与鉴权
- 通常从老板端跳转带 token 与 sid/sn/sname；或本地已有 token 时直接打开，再拉取 `GET /api/merchant/shops` 取当前店铺。
- 用于单店视角：只看该店的销售额、赔付、利润及该店订单。

### 5.2 数据加载
- `GET /api/shop/:shopId/orders`、`GET /api/draw/latest`。
- 本期订单：有已完成开奖时按 draw_id = latest.drawId 筛；无开奖时只算 status=paid。与老板端逻辑一致。

### 5.3 展示内容
- 开奖结果：1/2/3 奖号码（primer/segundo/tercero 或 winning_numbers 解析）。
- 销售额、Chance/Billete 销售额、总赔付、利润（销售额 - 总赔付）。总赔付 = 本期订单的 `getOrderWinAmount(order)` 之和。
- 已付款订单列表（可点开订单详情）；Billete/Chance 中奖号码汇总（从各单 win_breakdown 聚合）。
- 最近 7 期结算记录：`GET /api/settlement/history?shopId&limit=7`。

### 5.4 订单详情弹窗
- 请求 `GET /api/orders/:orderNumber`，展示订单号、金额、状态、中奖金额（getOrderWinAmount）、号码及 win_breakdown 明细。

### 5.5 期次与结算时间线（与用户确认）

| 时间点 | 销售页（老板端本期） | 结算页（result） |
|--------|----------------------|------------------|
| 上一期开奖后 16:00 | **清除已开奖期的销售数据**，开始统计**新一期**销售（订单属新期 draw_id） | — |
| 本期 14:55 | 停售 | — |
| 本期 15:00 前 | — | **清空当前期**（当前期转为历史），结算页空出，等待**本期**开奖后数据注入（不需定时任务，15:00 前清除即可） |
| 本期 15:00 | 开奖；本期数据在逻辑上**转入结算页**（开奖结果 + 订单结算注入） | 开奖后展示本期销售额/赔付/利润/订单（等总后台发送开奖结果后注入） |

- 清空的是**结算页**（当前期→历史），不是清空销售界面。销售页在 **16:00** 就清除已开奖期数据，以便开始售卖新一期。
- 实现上：不删库；当前期/历史用展示规则区分；**不需 9 点定时任务**，开奖当天 15:00 前清除结算页即可。历史 20 期 + 点进某期详情见《期次与结算逻辑评审》。
- **结算页订单列表**：按订单号后四位排序；样式与客户端一致——已中奖黄色，未中奖灰色。

## 六、后端接口与状态约定

### 6.1 订单状态
- `0` pending：未付款
- `1` paid：已付款，待开奖
- `2` settled：已开奖未中奖
- `3` won：已中奖（可兑奖）
- `-1` canceled：超时或取消  
前端可能同时兼容字符串 'pending'/'paid'/'settled'/'won'/'canceled'。

### 6.2 关键接口一览
- **订单**：`POST /api/orders` 下单；`GET /api/orders/:orderNumber` 查单；`POST /api/orders/:orderNumber/confirm` 确认收款；`POST /api/orders/:orderNumber/redeem` 兑奖。
- **店铺**：`GET /api/shop/:shopNumber` 按店号查店；`GET /api/shop/:shopId/orders` 店铺订单（支持 status、suffix 筛选）。
- **开奖**：`GET /api/draw/pending` 待开奖期；`GET /api/draw/latest` 最近开奖；`POST /api/draw/time` 设开奖时间；`POST /api/draw/manual` 手动开奖（会触发 settleOrdersForDraw）；`GET /api/draw/fetch-firebase` 拉 Firebase。
- **结算**：`POST /api/settlement/settle/:drawId` 期次结算（SettlementService）；`GET /api/settlement/history` 历史。
- **商户**：`POST /api/merchant/login`；`GET /api/merchant/shops`。
- **管理**：`POST /api/admin/assign-shop` 等（需 AdminToken）。

### 6.3 中奖规则（与数据库/协作说明一致）
- **Billete**：每奖只取最高档（4位全中→前3→后3→前2→后2→后1），三奖金额相加。赔率：exact [2000,600,300]，first3/last3 [50,20,10]，first2 [3,0,0]，last2 [3,2,1]，last1 [1,0,0]。
- **Chance**：只看一二三等奖的**后两位**；[14, 3, 2] 元/张，三奖可叠加；投注号统一取 `num.slice(-2)`。**结算时必须按「张数×赔率」**，不能按「金额×张数」。
- 订单级结算在 **DrawController.settleOrdersForDraw**（手动开奖时调用）；**SettlementService** 若也有 Chance 计算，必须使用 quantity 而非 betAmount，且 dist 与 src 一致（曾出现编译后 CHANCE 变成 80 导致全部按 80/张算错）。

### 6.4 订单与期次关系
- 下单时订单带上当前 `draw_id`（待开奖期）；若确认收款时订单尚无 draw_id，后端可补写为当前待开奖期。
- 开奖完成后，仅对「draw_id = 当前完成期且 status=1」的订单执行 settleOrdersForDraw，更新 win_amount、win_breakdown、status(2/3)。

## 七、查 Bug 时可重点核对的点

1. **两条结算路径**：DrawController 的 settleOrdersForDraw 与 SettlementService；两处对 Chance 的计算是否都按「张数×[14,3,2]」，有无 betAmount 误用。
2. **编译一致性**：部署的 dist 中 CHANCE 相关常量是否为 [14,3,2]，有无被优化/替换成 80 或其它值。
3. **本期订单与统计**：老板端/result 的「本期」是否都按同一规则（pending draw_id 或 draw_id 为空）筛选；开奖后热门号码是否只基于本期已付款。
4. **前端金额展示**：所有展示中奖金额的地方是否统一用 win_amount，为 0 时用 win_breakdown 之和，有无前端用赔率重算导致与后端不一致。
5. **订单状态与 draw_id**：确认收款、开奖、结算的时序；订单是否在正确的 draw_id 下被结算；兑奖校验 shopId、status=3、redeemed_at 为空。
6. **总后台开奖后**：手动开奖 → settleOrdersForDraw → 创建下一期 → 前端 initDrawDefaults 拉 pending 得到下一期日期；若 SettlementService 也在此链路上被调用，其逻辑与 DrawController 是否一致、是否重复写订单 win_amount。
